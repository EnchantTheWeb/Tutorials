Why Test Your Code?

1. Find Bugs Early
Tests help you catch mistakes before your code reaches users. It's much easier to fix a bug when you know exactly what went wrong!
2. Make Changes Confidently
When you update your code, tests verify everything still works. You won't accidentally break something!
3. Save Time Later
Writing tests takes time upfront, but saves WAY more time later. No more manual checking!
4. Help Other Programmers Understand Your Code
Tests show how your code is supposed to be used. They're like instructions for other developers.

Jest Basics JavaScript

What is Jest?
Jest is a testing framework for JavaScript. A framework is like a pre-built toolbox that makes it easier to write tests. Jest handles running your tests and telling you which ones passed or failed.
Setting Up Jest
First, you need to install Jest using npm (Node Package Manager). Think of npm as an app store for JavaScript packages!
Installation Command:
npm install --save-dev jest
ðŸ“Œ
What does this mean? This command downloads Jest and saves it as a "development dependency" - code you need while developing but not when your app runs.
Running Jest
Once installed, you can run your tests from the command line:
npm test

Pytest Basics Python

What is Pytest?
Pytest is a testing framework for Python. Like Jest, it makes writing tests easier by providing built-in tools and clear syntax. Python is an easy language for beginners, and Pytest is very beginner-friendly too!
Installing Pytest
Install Pytest using pip (Python's package manager):
Installation Command:
pip install pytest
Running Pytest
Run all tests in a folder or file:
pytest
# or run a specific file
pytest test_myfile.py

How to Structure a Test

The Basic Pattern: Arrange, Act, Assert (AAA)
Every test follows this simple 3-step pattern:
  1. Arrange: Set up the conditions for your test. Create objects and data.
  2. Act: Do the thing you want to test. Call a function.
  3. Assert: Check that the result is what you expected.
Jest Test Example
Here's a simple test in Jest that checks if adding two numbers works:
JavaScript with Jest:
function add(a, b) {
  return a + b;
}

test('adds two numbers correctly', () => {
  // ARRANGE: We have two numbers
  const result = add(2, 3);
  
  // ACT & ASSERT: Check that 2+3 equals 5
  expect(result).toBe(5);
});
ðŸ“Œ
Breaking it down:
  • test() tells Jest: "Here's a test"
  • The string describes what the test does
  • expect() is how we check if something is true
  • toBe() checks if the value equals exactly 5
Pytest Test Example
Here's the same test, but written in Python with Pytest:
Python with Pytest:
def add(a, b):
  return a + b

def test_adds_two_numbers_correctly():
  # ARRANGE: We have two numbers
  result = add(2, 3)
  
  # ACT & ASSERT: Check that 2+3 equals 5
  assert result == 5
ðŸ“Œ
Breaking it down:
  • In Pytest, test functions start with test_
  • assert is Python's built-in way to check if something is true
  • If the assertion is true, the test passes. If false, it fails.

Common Assertions

What are Assertions?
Assertions are statements that check if something is true. If it's not true, the test fails. Think of them as questions you ask: "Is this value equal to 5? Is this string empty?"

Jest Assertions

// Check if equal
expect(5).toBe(5);

// Check if includes
expect([1,2,3]).toContain(2);

// Check if true/false
expect(true).toBe(true);

// Check if defined
expect(value).toBeDefined();

// Check if null
expect(value).toBeNull();

Pytest Assertions

# Check if equal
assert 5 == 5

# Check if in list
assert 2 in [1,2,3]

# Check if true/false
assert True

# Check if not none
assert value is not None

# Check if none
assert value is None
ðŸ’¡
Pro Tip: Assertions are the "heart" of your tests. Always make sure you're checking something specific and meaningful!

Mocking & Test Doubles

What is Mocking?
Sometimes your code uses other functions, APIs, or databases. You don't want to call the REAL ones in tests (that's slow and risky). Instead, you use a "mock" - a fake version that does what you need for testing. It's like using a practice dummy instead of a real person!
Why Use Mocks?
  • Faster: Fakes are quicker than calling real APIs or databases
  • Safer: You won't accidentally delete real data or charge real credit cards
  • Reliable: Fakes work the same way every time, making tests predictable
  • Isolated: Test only your code, not other people's code
Simple Mocking Example (Jest)
// Let's say we have a function that calls an API
const fetchUserData = jest.fn();

// We tell the mock what to return
fetchUserData.mockResolvedValue({ name: 'Alice' });

// Now when our code calls fetchUserData,
// it gets our fake data instead!
fetchUserData(); // Returns { name: 'Alice' }
ðŸ“Œ
Key Concept: Mocking replaces real functions with fake ones for testing. You control what they return, making your tests predictable and fast!

Testing Best Practices

âœ¨ Golden Rules of Testing

  • Test One Thing Per Test: Each test should check ONE behavior. If it checks multiple things, it's hard to know what failed.
  • Use Clear Test Names: The test name should explain what it tests. Good: test_calculator_adds_positive_numbers() | Bad: test_it()
  • Keep Tests Simple: If your test is confusing, other programmers won't understand it. Keep the logic simple!
  • Test Edge Cases: Don't just test the "happy path" (normal usage). Test weird inputs: empty strings, negative numbers, null values.
  • Don't Test Library Code: You don't need to test code from other libraries. Trust that they tested it! Focus on YOUR code.
  • Make Tests Independent: Tests shouldn't depend on other tests. Each test should work by itself.
  • Run Tests Often: Run tests after every change. This catches bugs early!
What NOT to Test
Don't waste time testing:
  • Constructor methods (usually too simple)
  • Getters and setters (usually too simple)
  • External library code (they have their own tests)
  • Configuration files
Coverage Tips
"Code coverage" measures how much of your code is tested. Aim for 80% coverage, not 100%. Some code (like error handling for rare cases) is hard to test and not always worth it. Focus on testing the most important parts first!
